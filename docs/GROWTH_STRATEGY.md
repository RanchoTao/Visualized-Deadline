# Open Source Growth Strategy

This strategy is designed for an early-stage solo developer building Visualized-Deadline into a credible open-source product.

---

## Positioning

Most productivity projects compete on task management. VD should compete on **pressure visualization**.

The core public message:

> “This is not another todo app. It shows how deadlines, importance, relationships, recovery, and life domains create pressure.”

The first goal is not to convince everyone. The first goal is to make the right people instantly say: “I understand why this needs to exist.”

---

## Getting GitHub Stars

### Make the README star-worthy

- Put a strong banner at the top.
- Use one clear slogan.
- Show the product in the first screen with screenshots or GIFs.
- Explain the difference from todo apps with a comparison table.
- Add a roadmap that feels ambitious but credible.
- Make setup instructions short and reliable.

### Create a demo-first launch

- Pin the best GIF near the top of the README.
- Create a small demo dataset that shows pressure rising.
- Use a short launch post with a direct GitHub link.
- Share the same core visual repeatedly: “pressure field, not task list.”

### Make starring feel meaningful

Add language like:

> Star the repo if you want a local-first, pressure-aware alternative to traditional todo apps.

Do not beg for stars. Tie stars to the mission.

## Attracting Contributors

### Create contribution lanes

Not every contributor needs to be a core engineer. Label issues by contribution type:

- `good first issue`
- `design`
- `documentation`
- `accessibility`
- `pressure-engine`
- `graph-visualization`
- `local-first`
- `ai-safety`
- `plugin-ecosystem`

### Write issues as product problems

Bad issue:

> Improve tasks.

Good issue:

> Make high-pressure tasks visually distinct from low-pressure tasks in the daily control center.

Include:

- current behavior;
- desired behavior;
- why it matters;
- files likely involved;
- acceptance criteria;
- screenshot/GIF target if visual.

### Welcome non-code contributions

VD benefits from:

- screenshot composition;
- GIF storyboards;
- pressure model critique;
- UX writing;
- onboarding flows;
- graph layout ideas;
- local-first privacy reviews.

## Building Developer Trust

Trust comes from consistency and transparency.

Priorities:

- Keep `npm run build` working.
- Document local-first data behavior.
- Explain migrations carefully.
- Avoid unnecessary dependencies.
- Write security and privacy docs early.
- Keep roadmap promises staged.
- Use clear release notes.
- Respond respectfully to issues.

Make the project look maintained even before it is large:

- close stale experiments or move them to roadmap notes;
- keep README screenshots current;
- maintain a small set of high-quality issues;
- add checklists to PRs;
- use consistent labels.

## Creating Viral Demos

VD’s demos should show what a flat todo app cannot show.

Strong demo patterns:

1. A calm dashboard becomes high pressure as deadlines approach.
2. Completing one task causes the pressure field to visibly relax.
3. Ignoring recovery pushes the system toward overload.
4. A life map reveals that all pressure is coming from one domain.
5. A social graph shows a neglected relationship becoming a subtle signal.
6. An AI review summarizes pressure without shaming the user.

The visual story should be understandable without audio.

---

## Using GIFs Effectively

Good GIF rules:

- Keep each GIF under 15 seconds when possible.
- Show one concept per GIF.
- Add a visible cursor path or focus highlight.
- Use realistic demo data.
- Start with the final visual hook in the first 2 seconds.
- Avoid tiny text that cannot be read on GitHub mobile.
- Put the best GIF near the top of the README.
- Use captions that explain the transformation.

Recommended first GIF:

> “A deadline moves closer, pressure rises, VD recalculates focus, and the user completes one item to reduce overload.”

## Writing Good Issues

Every issue should make it easy for a contributor to succeed.

Template:

```md
## Problem
What pressure, structure, UX, or technical problem exists?

## Desired outcome
What should be true after this issue is solved?

## Suggested approach
Optional implementation or design direction.

## Acceptance criteria
- [ ] Clear testable result
- [ ] Documentation updated if needed
- [ ] Screenshot/GIF included for UI changes

## Context
Links, screenshots, related files, prior decisions.
```

## Maintaining Momentum as a Solo Developer

### Weekly rhythm

- Ship one small visible improvement.
- Improve one documentation page.
- Close or refine one issue.
- Post one screenshot, GIF, or devlog.

### Monthly rhythm

- Cut a tagged release.
- Update the roadmap.
- Record one demo GIF.
- Review security/privacy assumptions.
- Highlight one contributor or community suggestion.

### Avoiding burnout

VD is a pressure-aware product. The project should model the behavior it encourages:

- do not overpromise;
- keep roadmap phases realistic;
- preserve recovery time;
- celebrate small releases;
- prune low-value ideas;
- keep the core thesis sharp.

---

## Early Launch Sequence

1. Polish README hero and comparison table.
2. Add 3 screenshots and 1 strong GIF.
3. Add demo dataset.
4. Create 10 high-quality issues.
5. Tag `v0.10` or similar.
6. Post on GitHub, X/Twitter, Reddit, Hacker News, Product Hunt upcoming, and relevant Discords.
7. Ask for feedback on the pressure model, not vague “thoughts.”
8. Convert feedback into public issues.
9. Ship visible improvements weekly.
